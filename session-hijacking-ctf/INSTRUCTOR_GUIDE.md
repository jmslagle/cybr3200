# Session Hijacking CTF Challenges - Instructor Guide

## Overview

These four challenges teach session hijacking concepts through hands-on exploitation:

| Challenge | Difficulty | Time | Core Vulnerability | CWE |
|-----------|------------|------|-------------------|-----|
| 1. Predictable Tokens | Easy | 10-15 min | Weak token generation (base64 encoded user ID) | CWE-330 |
| 2. The Undying Session | Easy | 10-15 min | No session expiration / replay attack | CWE-613 |
| 3. Logout is a Lie | Easy-Medium | 10-15 min | Client-side only logout | CWE-613 |
| 4. Token Entropy Crisis | Medium-Hard | 30-45 min | Insufficient entropy, brute-forceable tokens | CWE-330, CWE-331 |

---

## Client-Server Trust Boundary Analysis

Each challenge should prompt students to articulate:

### Challenge 1: Predictable Tokens
- **Client controls:** Cookie value containing session token
- **Server trusts:** That tokens can only be generated by the server
- **Trust failure:** Token algorithm is deterministic and reversible; client can forge valid tokens
- **Key insight:** Security through obscurity (base64) ≠ cryptographic security

### Challenge 2: The Undying Session  
- **Client controls:** When to present the token
- **Server trusts:** That leaked tokens will eventually become invalid
- **Trust failure:** Server never invalidates old tokens; any valid token works forever
- **Key insight:** Token confidentiality is not sufficient; tokens need lifecycle management

### Challenge 3: Logout is a Lie
- **Client controls:** Cookie storage (delete, modify, replay)
- **Server trusts:** That deleting the client cookie terminates the session
- **Trust failure:** Session validity is determined by client-side state, not server-side state
- **Key insight:** Logout must invalidate server-side session state, not just client cookies

### Challenge 4: Token Entropy Crisis
- **Client controls:** Cookie value, ability to make unlimited requests
- **Server trusts:** That MD5 + timestamp + "random" provides sufficient entropy
- **Trust failure:** Small random space (1000 values) is trivially enumerable
- **Key insight:** Entropy must account for attacker's computational ability; 1000 guesses is nothing

## Deployment Instructions

### Prerequisites
- Docker installed on host system
- CTFd instance running
- Cloudflare Tunnel configured (per your existing setup)

### Building Containers

```bash
# Build all challenges
cd session-hijacking-ctf

docker build -t session-ch1-predictable ./challenge1-predictable-tokens/
docker build -t session-ch2-undying ./challenge2-undying-session/
docker build -t session-ch3-logout ./challenge3-logout-lie/
docker build -t session-ch4-entropy ./challenge4-entropy-crisis/
```

### Running Containers

```bash
# Run each on a different port
docker run -d --name ch1-predictable -p 5001:5000 session-ch1-predictable
docker run -d --name ch2-undying -p 5002:5000 session-ch2-undying
docker run -d --name ch3-logout -p 5003:5000 session-ch3-logout
docker run -d --name ch4-entropy -p 5004:5000 session-ch4-entropy
```

### Custom Flags (Optional)

Override default flags via environment variables:

```bash
docker run -d --name ch1-predictable -p 5001:5000 \
  -e FLAG="flag{your_custom_flag_here}" \
  session-ch1-predictable
```

### Cloudflare Tunnel Integration

Add routes in your tunnel config for each challenge:

```yaml
ingress:
  - hostname: predictable.yourdomain.com
    service: http://localhost:5001
  - hostname: undying.yourdomain.com
    service: http://localhost:5002
  - hostname: logout.yourdomain.com
    service: http://localhost:5003
  - hostname: entropy.yourdomain.com
    service: http://localhost:5004
```

Or use path-based routing through your existing setup.

## Expected Solutions

### Challenge 1: Predictable Tokens

1. Student logs in with any test account
2. Examines session_token cookie in Burp or browser dev tools
3. Recognizes it's base64: `dXNlcl8xMDAx` decodes to `user_1001`
4. Notices pattern: `user_<id>` where ID is shown on their profile page
5. Tries sequential IDs, finds admin is ID 1000
6. Forges token: base64("user_1000") = `dXNlcl8xMDAw`
7. Sets cookie manually, accesses /dashboard, gets flag

**Common Mistakes:**
- Not recognizing base64 encoding
- Trying to brute-force IDs from 0 instead of noticing admin is probably lower than regular users
- Forgetting to encode the forged token

---

### Challenge 2: The Undying Session

1. Student logs in, explores the application
2. Finds /incidents page with "resolved" security incident
3. Reads the incident report, notices leaked token: `a]48f!Kx9#mZq2$vL7@nR5`
4. Notes the incident was marked "resolved" without token rotation
5. Sets cookie `session_id=a]48f!Kx9#mZq2$vL7@nR5` in browser/Burp
6. Accesses /classified, gets flag

**Common Mistakes:**
- Not reading the incident details carefully
- Assuming the token format is wrong because it contains special characters
- URL encoding the token (not necessary for cookies)

---

### Challenge 3: Logout is a Lie

1. Student logs in as manager (who has admin panel access)
2. Captures auth_token cookie in Burp
3. Clicks "Secure Logout" - notes the reassuring messages
4. Observes cookie is deleted in browser
5. Manually re-adds the captured cookie (Burp Repeater or browser dev tools)
6. Accesses /admin - still works! Gets flag

**Deeper Investigation Path:**
1. Before logout, check /debug/sessions (shows N active sessions)
2. After logout, check again (count doesn't decrease for their session)
3. Recognizes server-side session was never invalidated

**Common Mistakes:**
- Not capturing the token BEFORE clicking logout
- Not knowing how to manually set cookies
- Trusting the logout confirmation messages

---

### Challenge 4: Token Entropy Crisis

1. Student logs in, examines their session cookie (MD5 hash)
2. Finds /audit logs page, reads the algorithm description:
   - `token = MD5(username + unix_timestamp + random_padding)`
   - random_padding range: 0-999
3. Finds admin login timestamp in logs: `1737450000`
4. Writes brute-force script:

```python
import hashlib
import requests

TARGET = "http://challenge-url"
TIMESTAMP = 1737450000

for rand in range(1000):
    token = hashlib.md5(f"admin{TIMESTAMP}{rand}".encode()).hexdigest()
    resp = requests.get(f"{TARGET}/vault", cookies={"session": token})
    if "Master Encryption Key" in resp.text:
        print(f"Found! rand={rand}, token={token}")
        print(resp.text)
        break
```

5. Runs script, finds valid token at rand=42
6. Gets flag from /vault

**Common Mistakes:**
- Missing the timestamp in the audit logs
- Not understanding how MD5 works
- Off-by-one errors in brute force range
- Not reading the algorithm description on the audit page
- Trying to brute-force the timestamp instead of the random component

## Assessment Notes

### For Written Analysis (if requiring writeups)

Strong submissions should:
1. **Identify the vulnerability** using proper terminology (CWE/OWASP)
2. **Explain the trust boundary failure** - what does the server assume that isn't true?
3. **Describe the exploitation steps** clearly
4. **Propose remediation** that addresses root cause:
   - Ch1: Use cryptographically secure random tokens (secrets.token_urlsafe)
   - Ch2: Implement token expiration and rotation after security incidents
   - Ch3: Invalidate server-side session on logout, not just client cookie
   - Ch4: Use sufficient entropy (256+ bits), don't roll your own token generation

### Oral Defense Questions

- "Why doesn't base64 provide security?" (Ch1)
- "What's the difference between authentication and session management?" (All)
- "Why isn't deleting the cookie sufficient for logout?" (Ch3)
- "How much entropy is 'enough' for session tokens?" (Ch4)
- "What OWASP Top 10 category do these fall under?" (A07:2021 - Identification and Authentication Failures)

## Variants for Future Semesters

### Challenge 1 Variants
- Use hex encoding instead of base64
- Use MD5(user_id) - still reversible via rainbow tables
- Use timestamp-based sequential tokens

### Challenge 2 Variants
- Token in server logs instead of incident report
- Token in error message
- Token in HTML comment

### Challenge 3 Variants
- Logout only works for some session types
- Session invalidation has a race condition
- Add "remember me" functionality that persists incorrectly

### Challenge 4 Variants
- Change timestamp to milliseconds (still enumerable with time window)
- Change random range to 0-9999 (still trivially brute-forceable)
- Add a small random offset to the timestamp (±10 seconds)

## Troubleshooting

### Container won't start
```bash
docker logs ch1-predictable
```

### Flag not matching
Ensure you're using the correct flag for the challenge. If you overrode flags, verify the environment variable was set correctly.

### Students can't reach the challenge
1. Verify container is running: `docker ps`
2. Test locally: `curl http://localhost:5001`
3. Check Cloudflare Tunnel status
4. Verify DNS propagation

### Challenge 4 solver doesn't work
- Verify the ADMIN_LOGIN_TIMESTAMP matches what's shown in audit logs
- Check that the container was built fresh (old builds might have different values)
